### four positions — direct/indirect duality + ontology-body metaphor (2026-04-08)

The four surviving software positions (make models, generate live data, gatekeep permissions, own atoms) each have a direct and indirect play. Direct = BE the thing (run the model, own the data, hold the permission, control the atom). Indirect = help others do the thing (RL company helping labs train, Alexandria helping users create live data AND hosting it).

Alexandria occupies both lanes of position two: indirect (scaffolding for users to create live data) and direct (hosting/serving that live data via Library).

**AGI = ontology + models + tools.** Alexandria is the ontology layer. Body metaphor: ontology is body, models are brain, tools are hand, the user is heart. Brain without body is a jar. Hands without body can't reach. The body is the connective tissue that makes the other three useful. Without structure, models hallucinate, tools flail, data is noise.

Tension: "scaffolding" implies temporary. If ontology is body, it's permanent infrastructure. Unless the claim is that Alexandria is scaffolding until models can build their own ontological understanding of each user — body becomes native to the brain. Stronger and scarier version of the thesis.

**Resolution: scaffolding for the individual, infrastructure for the network.** The server delivers two types of value. Scaffolding value (Blueprint methodology, setup, hooks) thins over time for any individual Author — Blueprint v47 is marginally better than v1. But the server IS the Library infrastructure — hosting the live personal data network (published shadows, payments, cognitive matchmaking, Factory signal). Network value compounds with every new Author. The server's total value to any individual Author may actually increase as the network grows, even as the scaffolding value thins. Free-first is structurally required because network effects need critical mass — not because the server becomes less valuable. Revenue comes from the network layer: Library cuts, premium content, social graph, aggregate opportunities.

This cleanly maps to the direct/indirect duality: indirect play (helping users create live data) is scaffolding — thinning per-user. Direct play (hosting the network of everyone's live data) is infrastructure — compounding across all users. The scaffolding thins for each user. The infrastructure compounds across all users.

*Status: crystallised 2026-04-09. Migrated to a3 (WHY FREE-FIRST IS STRUCTURALLY REQUIRED section in revenue model), a4 (Library as collective asset section strengthened with scaffolding/infrastructure duality and direct/indirect mapping), Memo.md (four positions paragraph expanded with both sides of position two). The ontology-body metaphor and the tension ("scaffolding implies temporary") remain developing — the resolution (scaffolding for individual, infrastructure for network) is crystallised but the scarier version (body becomes native to brain) is parked.*

### Mythos as positioning tailwind (2026-04-08)

Anthropic launched Claude Mythos Preview — a frontier model purpose-built for finding zero-day vulnerabilities and writing functional exploits autonomously. Found thousands of unpatched critical bugs across every major OS and browser. Restricted to 12 vetted security partners (Apple, Microsoft, Amazon, etc.) via Project Glasswing.

**The panic:** people are saying airgap everything, cloud is dead for sensitive data. Any server with valuable data will eventually be found and exploited. The capability asymmetry during the transition period (model finds bugs faster than humans patch them) makes this existential for anything that stores user data centrally.

**Why this is our thesis validated by our own parent company:**
- Alexandria's architecture was built on philosophical grounds: your self-knowledge should be sovereign, local, yours.
- Mythos just turned that philosophical position into a security argument. We didn't have to pivot — the world moved toward us.
- A complete server breach of Alexandria yields: hashed API keys, encrypted metadata, and anonymous usage patterns. No constitutions, no vaults, no self-knowledge, no cognitive data. There is literally nothing to steal.
- "Cloud is dead for personal data" is not our claim. It's the inevitable conclusion once Mythos-class capability proliferates.

**What we shipped in response (same day):**
1. API keys hashed server-side (SHA-256) — server never stores raw keys
2. Accounts blob encrypted at rest (AES-256-GCM) — KV breach yields encrypted data without Worker env key
3. Raw API keys removed from ALL emails and Stripe metadata — key shown once on callback page only
4. `DELETE /account` endpoint — full data deletion across KV, D1, R2, Stripe
5. Ed25519 Blueprint signing — private key on founder's laptop, public key in user's hooks. Compromised server can't forge methodology.
6. Immutable hooks — no auto-update of shell scripts. Server can never push code to user machines.
7. Canary instructions in signed Blueprint — AI rejects and reports suspicious instructions from unsigned sources.

**The structural claim (true at any hacker quality):**
- Cognitive data (constitution, vault, self-knowledge) → structurally unhackable. Not on server.
- Methodology (Blueprint) → signed. Server compromise can't forge.
- Code execution (hooks) → immutable. Server can't push.
- Metadata (emails, usage patterns) → encrypted + hashed, low-value if leaked.
- Factory delta → unsigned, lower trust, canary-guarded. The soft point — but it can only suggest, not override.
- The only machine that can compromise users: the founder's laptop (signing key).

**The pitch (investor angle):**
"Alexandria stores your self-knowledge on your machine. A complete server breach yields nothing — hashed credentials, encrypted metadata, no ability to push code or modify methodology. We didn't build this because of Mythos. We built it because the philosophy demands it. Mythos proved the philosophy was right."

**The irony:** Anthropic's security research proves Alexandria's architecture thesis. The company that makes our intelligence layer just validated our product positioning.

### daily email — the morning brief (2026-04-08)

The nudge problem: Alexandria has zero visibility between sessions. Start hook nudges never fire because users don't keep terminal tabs open. The founder — user 0 — has never once seen a nudge. Email is the only push channel that meets users where they already are.

**The product move:** not "here's what your AI did overnight." It's "here's what you should think about today." The gym trainer texting today's workout. Notification is SaaS. Practice is Alexandria.

**Architecture:**
- Autoloop trigger already runs daily → at the end, generates brief + one prompt → stores in KV or hits email endpoint
- Resend (already in stack) delivers. Minimal HTML — Claude email aesthetic, not SaaS
- The prompt is one line, copy-pasteable into any chat. Engine-generated from current Machine state, not templated. Bitter lesson: model picks the prompt
- Deep link to open terminal is a later optimisation

**Frequency is an intelligence decision, not a hard-code:**
- Soft default: daily or every-other-day (TBD — needs data)
- Machine adjusts per-Author as it learns engagement patterns (opens, click-throughs, session-starts-after-email)
- The model picks the frequency. Not a setting with three radio buttons — a continuous intelligence decision

**Privacy — zero exposure of inner state:**
- Email can reference what the *system* did ("your constitution was updated"), never what the *person* thinks or feels
- No constitution content, no vault content, no ontology content in email body
- The prompt is forward-looking ("explore X today"), never backward-looking ("you said Y yesterday")

**The unsubscribe signal is a Factory loop:**
- "Send less" / "send none" (or whatever the right language is — TBD)
- Signal feeds into Blueprint → auto-adjusts default frequency for new Authors
- Each Machine also learns individually from its Author's engagement
- The meta-loop watches the loop: if 60% of Authors dial down, default shifts

**Timing is an intelligence decision:**
- Soft default: 9am SF (Pacific). Machine adjusts per-Author as it learns when emails convert to sessions
- Requires timezone at signup (infer from GitHub location, or ask)

**Three components, each doing different work:**
1. **Brief** — what happened overnight (backward-looking). Factual system delta. "3 vault fragments processed. constitution updated." Never content, never interpretation — just what the system did
2. **Prompt** — notepad contents (forward-looking). How many fragments are parked, and what they are by topic label. "2 fragments in your notepad: stoic indifference vs christian compassion, the talent stack idea from Naval." The Author's own signal reflected back as a reminder, not the Machine's interpretation. The notepad is already forward-looking (parked for next session) and the topics are the Author's own words
3. **Nudge** — rotating quote + `/a`. Aristotle, Socrates, die empty, etc. The nudge library grows in Blueprint, Machine picks which one resonates for this Author

**Privacy principle:** the email shows what the *system* did (brief) and what the *Author* said (notepad topics). Never what the Machine *thinks* about the Author. Topic labels are the Author's own words — not an exposure, a reminder.

**Open questions:**
- What if autoloop had nothing to report and notepad is empty? Skip the whole email (let frequency intelligence handle it) or send nudge-only?
- Email design reference pass needed — match existing Alexandria email aesthetic (EB Garamond, #f5f0e8/#3d3630, centered, minimal)

*Status: developing. Next: build the Resend integration into autoloop trigger, spec notepad-to-email pipeline.*

### prosumer sales positioning — why not DIY, who doesn't qualify, how to sell (2026-04-09)

**The shadow is public information at scale.** The shadow contains nothing beyond what you'd share if a stranger talked to you long enough. It's a neo-biography — the digital equivalent of writing one book about how you think and putting it in a library. Not a paid public library, a members-only library. You're an Author contributing one book. That's the ask.

**DIY people are the easiest customers.** They're already sold on the concept. Already bought the frame. Don't need the philosophical pitch. Just need marginal value: we optimise the system continuously, it compounds, there are synergies and community. Pure marginal value proposition. The pitch to DIY people is shorter and easier than to anyone else.

**The opportunity cost argument.** It's not a one-time setup — it's continuous optimisation. Alexandria inherently tries to make your time more valuable, which means your opportunity cost goes up. Spending time fiddling with the system yourself is increasingly expensive. We ride the same tools you'd use. We just make the system compound without you maintaining it.

**Who doesn't qualify (the exclusion set is tiny):**
1. Security paranoids — addressed by sovereign architecture, signed Blueprint, immutable hooks, canary. The Mythos validation.
2. Loner penny-pinchers — can't refer 5 kin AND won't pay $10. Both conditions must hold.
3. Total recluses — people who literally never share any information with any stranger ever. The shadow is just what you'd say in conversation. Only a true recluse would object.
4. Brand haters — people who want the company to fail. Niche by definition.

**The sell — three angles:**
1. *Think better.* Mental gym. Humans are still relevant, but you need to optimise your thinking. Invest in yourself. The "underclass" thing is a mindset — don't surrender your cognition, sharpen it.
2. *Prove it.* If you think you're valuable, invest in that. Do a complete 180 — lean into the technology to optimise your thinking instead of letting it replace it.
3. *The Library provides clear marginal value, not just obligation:*
   - Earn money from your published shadow and works
   - Build your personal brand / cognitive identity
   - Meet and learn from other Authors (social network on the root-node level — cognitive compatibility, not demographic)
   - Easier path to free subscription (kin from the network)
   - Aggregate purchasing power — partner with companies who want access to this pool of minds, enabling better deals on goods/services that only work at aggregate level
   - "If you're interesting, people should be able to engage with your thoughts and pay for it. Same as buying someone's book."

**Root-node connection.** Every other social network connects on downstream/external dimensions (what you look like, where you went to school, what you clicked). Alexandria connects on the root node — how you actually think. Structurally different from everything that exists. Human-to-human connection will be critical with ai — this is the cognitive layer.

**The book analogy.** The shadow IS a biography. If you're interesting, people buy your book. This is the same thing, but living, and the library is curated. Contributing one book to the library is the minimum ask. It's what you'd share anyway — just at scale.

**The Four Constants.** Alexandria has exactly four non-negotiable requirements. Everything else is a variable — hyper-personalizable software the Author controls. The constants: pay, create data file, keep it live, publish one file to Library. Everything else — Engine behavior, constitution structure, Library page design, session style, machine setup — is the Author's to customize. This is why DIY people are ideal: Alexandria doesn't compete with their system. It's pure marginal value on top. The design.md metaphor: if you already have one and we ship one, you extract the delta and add it to yours. More money is more money. You don't say no.

**Users run RL for Alexandria.** Maximum customizability is not generosity — it is the learning architecture. Every Author's customization choices (Library page, machine setup, session preferences, publishing decisions) are exploration episodes. The Factory aggregates them. The Blueprint improves. The more freedom Authors have, the richer the signal. This is the same dynamic as the Library: the product is genuinely more valuable with more people making more choices.
